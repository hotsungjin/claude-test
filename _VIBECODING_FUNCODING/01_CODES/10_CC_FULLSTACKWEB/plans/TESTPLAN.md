# 테스트 계획서 (TESTPLAN.md)

## 🎯 테스트 전략
- **TDD (Test-Driven Development)** 방식으로 테스트를 먼저 작성
- 모든 예외 케이스와 엣지 케이스 포함
- API 테스트와 UI 테스트 병행
- 회귀 테스트를 통해 기존 기능 정상 작동 확인

## 📋 인증 기능 테스트 케이스

### 1. 회원가입 (/api/auth/signup)

#### ✅ 정상 케이스
- [ ] TC1-1: 유효한 이메일, 사용자명, 비밀번호로 회원가입 성공
- [ ] TC1-2: 회원가입 후 자동 로그인 (쿠키 설정 확인)
- [ ] TC1-3: users 테이블에 데이터 저장 확인 (is_admin=false 기본값)

#### ❌ 예외 케이스
- [ ] TC1-4: 중복 이메일로 회원가입 시도 → 409 에러
- [ ] TC1-5: 중복 사용자명으로 회원가입 시도 → 409 에러
- [ ] TC1-6: 잘못된 이메일 형식 → 422 에러
- [ ] TC1-7: 비밀번호 6자 미만 → 422 에러
- [ ] TC1-8: 비밀번호 확인 불일치 → 422 에러
- [ ] TC1-9: 필수 필드 누락 → 422 에러
- [ ] TC1-10: SQL Injection 시도 → 정상 처리 (Supabase가 방어)
- [ ] TC1-11: XSS 시도 (스크립트 태그 포함) → 정상 저장 (출력 시 이스케이프)

### 2. 로그인 (/api/auth/login)

#### ✅ 정상 케이스
- [ ] TC2-1: 이메일/비밀번호로 로그인 성공
- [ ] TC2-2: access_token, refresh_token 쿠키 설정 확인
- [ ] TC2-3: HttpOnly, Secure, SameSite 속성 확인

#### ❌ 예외 케이스
- [ ] TC2-4: 존재하지 않는 이메일 → 401 에러
- [ ] TC2-5: 잘못된 비밀번호 → 401 에러
- [ ] TC2-6: 이메일 형식 오류 → 422 에러
- [ ] TC2-7: 필수 필드 누락 → 422 에러
- [ ] TC2-8: 너무 많은 로그인 시도 → Rate limiting (Nginx 레벨)

### 3. 이메일 중복 체크 (/api/auth/check-email)

#### ❌ 예외 케이스

- [ ] TC3-4: 이메일 파라미터 누락 → 422 에러

### 4. 사용자명 중복 체크 (/api/auth/check-username)
